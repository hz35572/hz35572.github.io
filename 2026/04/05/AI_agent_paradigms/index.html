AI Agent 经典范式解析

AI Agent 经典范式解析（一）：概念与选型

大模型（LLM）很能“聊天”，但要真正在业务里落地，需要使用工具、处理多步任务、并在试错中学会改进。这些能力不是靠“一句 prompt”搞定，而是靠一套 Agent 范式来组织。

不同范式决定了智能体“什么时候想、什么时候做、什么时候改”。本文将详细拆解业界最经典的三大范式：ReAct、Plan-and-Solve 和 Reflection。

一、Agent 能力光谱总览

我们可以把三大范式放在一个“能力光谱”里理解：

mindmap
  root((LLM Agent 设计模式))
    ReAct（边想边做）
      推理与动作交替
      强依赖工具与环境
      适合决策任务（QA、游戏、Web 操作）
    Plan-and-Solve（先规划再执行）
      全局计划 + 子任务执行
      提示工程友好
      适合数学、多步推理、工作流编排
    Reflection（反思式学习）
      事后反思 + 语言反馈
      记忆/经验库
      适合复杂推理、代码生成、长期策略

二、ReAct：Reasoning and Acting（边想边做）

来源论文：ReAct: Synergizing Reasoning and Acting in Language Models（Yao et al., ICLR 2023）

2.1 核心思想

让 LLM 在同一个 Prompt 里交替生成：

  • Thought（思考过程）：推理、规划、诊断异常；
  • Action（具体动作）：搜索、查库、调 API；
  • Observation（环境反馈）：工具返回的结果。

简单讲：一边思考，一边动手，用动作的反馈修正下一步思考。

2.2 运行循环

flowchart LR
  Q["用户问题"] --> T1["Thought 1: 分析问题"]
  T1 --> A1["Action 1: 调用工具"]
  A1 --> O1["Observation 1: 环境返回"]
  O1 --> T2["Thought 2: 修正思路"]
  T2 --> A2["Action 2: 继续调用"]
  A2 --> O2["Observation 2"]
  O2 -->|足够信息| Ans["输出最终答案"]
  O2 -->|需更多信息| T3["Thought 3..."]

2.3 Prompt 示意

2.4 优缺点与场景

  • 优点：解释性极强；工具利用能力好；支持实时纠错。
  • 缺点：Token 消耗大、延迟高；对工具返回格式要求严格。
  • 适用：多跳问答、需要频繁调 API 的业务智能体。

三、Plan-and-Solve：先规划，再执行

3.1 核心思想

普通 CoT（Chain-of-Thought）是“一次性线性推理”，容易丢步或算错。Plan-and-Solve 将过程拆分为两步：

  1. Plan 阶段：把问题拆成若干子任务；
  2. Solve 阶段：按计划逐步执行。

3.2 进阶版：PS+ Prompting

在 PS 基础上加入细致指令，例如：“先提取变量”、“关注中间计算准确性”、“指定输出格式”，显著减少计算和遗漏错误。

3.3 Prompt 示意

3.4 优缺点与场景

  • 优点：提示工程极友好；显式减少“走偏”；易与工具结合。
  • 缺点：一次性全局规划在高度不确定任务中容易过时。
  • 适用：数学推理、多步 QA、业务流程编排。

四、Reflection / Reflexion：反思式学习

4.1 核心思想

传统 RL 需要大量样本+微调，而 Reflexion 选择：不更新权重，只更新记忆。
当任务失败时，让 LLM 用自然语言反思“哪里错了、为什么错了”，存入记忆库，下次尝试时作为上下文注入。

4.2 关键组件与循环

flowchart LR
  Q[任务] --> A1[Actor 生成行动]
  A1 --> E[环境执行]
  E --> R{成功?}
  R -->|是| END[结束]
  R -->|否| RF[Self-Reflection: 生成反思文本]
  RF --> M[Episodic Memory 保存]
  M --> A2[Actor 结合反思再次尝试]

4.3 优缺点与场景

  • 优点：对闭源 API 友好；反思过程可读可调试；成功率随轮次提升。
  • 缺点：成本与延迟极高；反思质量依赖模型能力。
  • 适用：代码生成、复杂推理、需要经验积累的长周期任务。

五、选型指南：什么时候用哪个？

维度 ReAct Plan-and-Solve Reflection
核心循环 想→做→看→想… 规划→子任务1→子任务2… 尝试→失败→反思→再尝试…
是否需要工具 强依赖 可有可无 可有可无
是否显式规划 局部规划 全局显式规划 偏事后反思
Token 成本 中高 中 高
最佳场景 工具多、交互多 步骤多、怕丢步 允许失败、要进化

生产环境黄金组合：Plan-and-Solve (定大局) + ReAct (做细节) + Reflection (纠错误)。

AI Agent 经典范式解析（二）：工程实现与架构设计

在第一篇中，我们理解了 ReAct、Plan-and-Solve 和 Reflection 的理论。但在生产环境中，“能跑通 Demo”和“能扛住线上流量”之间隔着巨大的工程鸿沟。

本文将从系统架构的角度，拆解这三种范式落地时的核心工程要点。

一、ReAct 的工程实现要点

ReAct 的核心难点在于：如何稳定地解析 LLM 的自由文本输出，并安全地与外部工具交互。

1. 工具集的标准化定义

不要在 prompt 里用自然语言模糊描述工具，必须使用结构化 Schema（类似 OpenAI Function Calling 的 JSON Schema）。

  • 明确 name、description、parameters（类型、枚举值、必填项）。
  • 为工具设定超时时间和熔断机制。

2. 输出格式的强制约束

经典 ReAct 依赖正则提取（如 Action[Search["xxx"]]），但在生产中极不稳定。工程改进方案：

  • 推荐：开启 LLM 的 function_call 或 tool_choice 模式，让模型直接输出 JSON；
  • 兜底：保留正则解析，但加入强大的异常捕获，解析失败时用预设话术要求模型重试。

3. 循环控制与防雪崩

  • 最大步数限制：强制设定（如最多 5 步 Action），防止死循环刷爆 Token；
  • 并行调用：如果 Thought 中包含多个独立的 Action，可设计 DAG 图并行执行（例如同时查天气和查航班），大幅降低延迟；
  • 上下文裁剪：超过 3 轮后，前面的 Observation 可能会撑爆上下文窗口，需做摘要或截断。

二、Plan-and-Solve 的工程实现要点

Plan-and-Solve 的难点在于：两阶段（规划与执行）的解耦与状态流转。

1. 两阶段编排器

不要试图用一个 Prompt 完成“规划+执行”。

  • **Stage 1 (Planner)**：专门的小模型/便宜模型负责将需求拆解为 JSON 格式的任务列表（[{"task_id": 1, "desc": "xxx", "depends_on": []}]）；
  • **Stage 2 (Executor)**：遍历任务列表，依次（或按依赖关系）调用大模型或工具执行。

2. 动态规划

一次性的静态计划在遇到异常（如 API 报错）时会卡死。

  • 引入 异常重规划机制：当某个子任务执行失败时，将错误信息传回 Planner，让它只重新生成该节点及其后续节点的计划。

3. 状态上下文传递

子任务 2 往往需要子任务 1 的结果。

  • 在系统中设计一个 Context Variables 黑板机制，每个子任务执行完毕后，将输出写入黑板，后续子任务的 Prompt 自动拼接黑板内容。

三、Reflection / Reflexion 的工程实现要点

Reflection 的难点在于：如何评估失败、如何存储反思、以及如何控制多轮成本。

1. Evaluator（评估器）的设计

不要让 LLM 自己评判自己是否成功（容易产生幻觉）。

  • 强规则评估：代码执行看 Exit Code；SQL 执行看是否报错；数学题用 Python eval() 算结果；
  • LLM-as-a-Judge：仅在主观任务（如文案质量）中使用，且必须提供极其详细的 Scoring Rubric。

2. Episodic Memory（情景记忆）管理

反思文本不能无限累加，否则会淹没原始任务。

  • 短期记忆：当前任务的 1~2 次反思，直接拼在 Prompt 里；
  • 长期记忆：历史的反思，存入向量数据库。下次遇到相似任务时，通过 Embedding 检索 Top-K 条反思注入。

3. 成本控制策略

Reflection 极其耗钱，必须有“止损线”：

  • 设定最大尝试轮数（如 max_attempts = 3）；
  • 退火机制：第 1 次失败用 GPT-4 反思；第 2 次失败用 GPT-3.5 反思；第 3 次直接返回错误，不再尝试。

四、统一架构蓝图

在复杂业务中，这三种范式往往融合在一个 Agent 系统中。推荐以下微服务架构分层：

graph TD
User[用户请求] --> Gateway[API 网关 & 鉴权]
Gateway --> Orchestrator[核心编排引擎]

subgraph 编排层
    Orchestrator --> Planner[Plan-and-Solve 模块]
    Planner --> ReActExecutor[ReAct 执行模块]
    ReActExecutor --> ToolEnv[工具沙箱环境]
end

ToolEnv -->|失败/成功| Evaluator[评估模块]
Evaluator -->|失败| ReflectionModule[Reflection 反思模块]
ReflectionModule -->|修改计划| Planner

subgraph 基础设施层
    ToolEnv
    Evaluator
    ReflectionModule --> VectorDB[(向量记忆库)]
    Orchestrator --> PromptMgr[版本化 Prompt 管理]
    Orchestrator --> ContextDB[(状态与上下文存储)]
end

这套架构做到了：规划解耦、执行标准、反思可控。

AI Agent 经典范式解析（三）：Python 实战代码精讲

前面讲了很多理论，本文我们直接上手。为了让你看清本质，我不使用 LangChain 等重封装框架，而是用最基础的 openai Python SDK，实现这三种范式的最小可行性代码（MVP）。

0. 准备工作：Mock 工具与环境

为了演示，我们先定义一个假工具和假评估器。

一、ReAct 的原生实现

核心逻辑：用正则解析 Action[输入]，循环调用，直到出现 Final Answer[结果]。

二、Plan-and-Solve 的原生实现

核心逻辑：第一次 API 调用生成 Plan（纯文本），第二次 API 调用带入 Plan 让模型 Solve。

三、Reflection / Reflexion 的原生实现

核心逻辑：尝试 -> 评估器报错 -> 将错误和原问题交给 LLM 反思 -> 带着反思结果重新尝试。

四、工程化延伸建议

上述代码是教学级别的 MVP。在你的真实项目中，需要做以下替换：

  1. 废弃正则提取：使用 OpenAI 的 tools / function_calling 参数，强制模型以 JSON 格式输出 Action，解析成功率从 90% 提升到 99.9%。
  2. 异步并发：在 ReAct 中，如果连续出现两个 Search[...]，应使用 asyncio.gather() 并发请求工具，响应时间减半。
  3. 状态持久化：如果 Reflection 需要跨会话记忆，需将 reflection_text 写入 Redis 或 Milvus 等向量数据库，下次查询相似任务时 RAG 检索出来。

总结

看懂这三段代码，你就掌握了当前市面上 90% Agent 框架（LangChain Agent、AutoGPT、AgentGPT）的底层底座：

  • ReAct = while 循环 + 正则/JSON提取 + 工具调用；
  • Plan-and-Solve = Two-pass Prompt 机制；
  • Reflection = Try-Except 机制 + 自然语言日志拼接。

框架会过时，但范式是永恒的。
