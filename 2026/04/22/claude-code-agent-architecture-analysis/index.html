2. 上下文管理：AI Agent的核心挑战

2.1 根本矛盾

AI Agent面临的核心矛盾是：需要超长上下文才能完成复杂任务，但超长上下文的Token成本会让产品在商业上不可持续。

具体表现：

  • 一轮长对话可能有100K+ Token的tool_result
  • 每轮对话都全量重发 → 成本飙升（$1+一轮）
  • 传统做法「删旧消息」→ prompt cache失效 → 重发全部 → 雪上加霜
  • 不删 → 上下文溢出，对话中断

2.2 System Prompt的两层缓存架构

Claude Code将Prompt拆分为两层，通过 分界线（BOUNDARY MARKER） 实现完全解耦：

维度 静态前缀（scope: ‘global’） 动态部分（scope: null）
内容 身份定义、系统规则、任务规范 会话指引、记忆、环境信息
缓存范围 跨用户、跨组织共享 每个会话独立
失效时机 几乎永不失效 /clear或/compact时失效
大小估计 5-8K Token 5-10K Token

技术经验：

  • 区分”永远不变的”和”每次不同的”，分别对待
  • 分界线确保动态部分变化时，静态部分的缓存绝对不会失效
  • 静态部分跨数百万用户共享，产生乘数效应

2.3 四层递进的Compact架构

当上下文接近上限时，Claude Code采用四层递进的压缩策略：

2.3.1 Cache Edits黑科技

这是Anthropic API的未公开功能，核心是缓存不失效的增量删除：

工作原理：

  1. 给每个块贴上ID（cache_reference）
  2. 告诉服务端”跳过这些块”（数据还在缓存里，只是被标记为”跳过”）
  3. 保存标记指令，供后续请求使用

优势：

  • 缓存前缀没变 → 下一轮还是命中
  • 模型看到的上下文变短了 → 省token
  • 原始数据还在服务端 → 只是被标记为”跳过”

2.3.2 SessionMemoryCompact：外科手术式的精确切割

核心规则：

  • 绝不拆散tool_use和tool_result — 工具调用和结果必须成对出现
  • 绝不拆散thinking块和同message.id的tool_use — 思考过程和执行动作必须在一起
  • 保留最近10K-40K token的消息（根据上下文压力动态调整）
  • 用一条Session Memory替代旧对话

技术经验：

  • 切割逻辑是”宁可多留一点，也不能拆散”的保守策略
  • 确保上下文的语义完整性

2.4 上下文压缩的五种策略

策略 触发条件 核心手段 代价
microcompact 基于时间（60分钟） 清理旧工具结果 最小
context collapse 90%/95%阈值 投影折叠（不修改原消息） 小
session memory token增长+工具调用阈值 用summary替代旧对话 中
full compact 接近上下文上限 Fork agent做全量摘要 较大
PTL truncation 极端情况 丢弃最早消息组 最大

技术经验：

  • 能轻量解决的不用重量级方案
  • 压缩后需要自动恢复关键上下文（最近文件、激活的Skills等）
  • 压缩不是单一的，而是根据上下文压力逐级升级

3.4 Memory写入的两条路径

3.5 AutoDream：离线记忆巩固

触发条件：

  • 距离上次 ≥ 24小时
  • 累积 ≥ 5个新session
  • 获取文件锁（防并发）

四个阶段：

  1. Orient（定向）：扫描已有记忆，了解当前知识库状态
  2. Gather（采集）：从日志和会话转录中提取新知识
  3. Integrate（整合）：合并新旧知识，修正矛盾，去重
  4. Prune（修剪）：精简索引，删除矛盾条目，优化记忆结构

技术经验：

  • 记忆不是单一的，而是分层的
  • 从”记录对话”到”学习知识”的自我进化
  • 每次交互 → 提取记忆 → 定期整合 → 更新认知 → 指导下一次交互

3.6 Memory主动召回机制（Active Recall）

完整召回流程：

  1. 扫描memory目录（最多200个文件）
  2. 去重过滤（alreadySurfaced集合）
  3. 构建候选清单
  4. Sonnet sideQuery打分选择（≤5个最相关）
  5. 读取完整内容+注入
  6. 更新alreadySurfaced

技术经验：

  • 用Sonnet而不是Opus：检索排序任务，Sonnet更快更便宜
  • 只看frontmatter摘要：不读完整文件内容来匹配
  • 同一条记忆在一个会话中最多被注入一次，避免浪费token

4. 多Agent协作：从单兵作战到分布式军团

4.1 为什么需要多Agent？

单个Agent有三个天然瓶颈：

  • 上下文污染 — 一个任务的中间结果会污染主对话历史
  • 无法并行 — 只能串行执行，效率低
  • 无法专业化 — 一个Agent要处理所有类型的任务

4.2 三种协作模式

模式 特点 适用场景 缓存
Fork 继承父上下文，缓存最优 调查、研究 共享父级cache
Subagent 专家顾问，独立上下文 需要独立视角/特定角色 独立缓存
Swarm 持久化团队，分布式协作 多Agent长期协作 各自独立

4.3 子Agent共享Prompt缓存

最关键的发现：当Claude Code fork一个子Agent时，它会创建一个与父上下文字节级完全一致的副本。而API会对这个上下文做缓存。

结论：同时启动5个Agent处理不同任务，成本几乎和1个Agent顺序执行差不多。

技术经验：

  • Fork直接复制父级渲染后的system prompt字节，避免Feature Flag冷→热切换导致缓存失效
  • 这是Cache感知的设计 — 连Agent的创建方式都要考虑缓存命中率

4.4 分布式权限管理

Swarm最精妙的设计是权限冒泡：

技术经验：

  • 多个Worker的权限请求汇总到Leader，用户只需审批一次
  • 即使在同进程，也用文件邮箱，保证三种运行方式的行为一致

4.5 子Agent的三种执行模型

模型 描述 缓存
fork 继承父上下文，缓存最优 共享
teammate 在tmux或iTerm中开独立面板，通过文件”邮箱”通信 独立
worktree 每个Agent一个独立git worktree，各自分支隔离 独立

5. 工具调用与安全控制

5.1 工具调用流程

5.2 智能批处理执行

类型 工具 执行方式
concurrent（并发） 只读操作（读文件、搜索、glob匹配） 最多10个并发
serial（串行） 有修改行为的操作（编辑、写入、bash命令） 逐个执行，避免冲突

5.3 五层权限配置体系

三种权限模式：

  • bypass — 完全不做权限检查（危险但极快）
  • allowEdits — 自动允许工作目录内的文件编辑
  • auto — 每个操作由一个LLM分类器判断（最平衡）

5.4 Bash安全分类器：20+道检查流水线

核心检查：

  • Shell元字符检测 — $() `` {} <> 等
  • 命令替换模式 — <() >() =() $( 等
  • 危险变量检测 — $IFS、$PATH 等
  • 输入/输出重定向检测
  • 花括号展开检测 — {a,b,c} 绕过
  • 反斜杠转义运算符 — \; \| \& 等
  • 引号反同步 — "x\"y" \; echo ~/.ssh/id_rsa
  • 控制字符 + Unicode空白字符
  • jq系统函数检测 — system()/exec/...
  • /proc文件系统访问 — /proc/self/environ 等

Zsh专属防御：

  • zmodload → 加载危险模块
  • emulate → eval等价物
  • zpty → 伪终端执行命令
  • ztcp → TCP连接外泄数据
  • zf_rm → 绕过rm权限检查

技术经验：

  • 每个检查都有唯一的checkId（1-26），触发时上报遥测
  • 沙箱执行 — 危险操作强制沙箱
  • 细粒度文件权限 — 敏感文件只读
  • 网络隔离 — 只允许特定域名，API调用白名单

7. 性能优化与缓存策略

7.1 Cache感知设计

Claude Code的每个设计决策都在考虑缓存命中率：

设计 缓存优化
System Prompt两层架构 静态前缀跨用户共享
Fork子Agent 直接复制父级渲染后的字节
Cache Edits 增量删除，保持缓存命中
MCP Instructions 使用DANGEROUS_uncachedSystemPromptSection
Auto Compact Fork摘要agent复用主对话的prompt cache

7.2 基础设施感知

Agent根据以下基础设施约束调整行为：

  • 缓存过期时间（5分钟TTL）
  • Token预算
  • 终端焦点状态

技术经验：

  • 让Agent理解基础设施的约束，而不是盲目执行
  • 这是工业级AI Agent和玩具Demo的区别

7.3 流式架构

  • 整个执行链路基于async generator
  • 每一步都会以事件流的形式逐步输出
  • 按下Escape，可以干净地中断当前响应
  • 之前的上下文保留，被打断的输出被干净丢弃

10.2 最佳实践

10.2.1 CLAUDE.md的使用

错误用法：

  • 空文件
  • 写成一本”圣经”

正确用法：

  • 把架构决策写进去
  • 文件约定写进去
  • 测试模式写进去
  • “绝对不要这样做”的规则

层级结构：

  • ~/.claude/CLAUDE.md — 全局（编码风格、偏好）
  • ./CLAUDE.md — 项目级（架构决策、约定）
  • .claude/rules/*.md — 模块化规则
  • CLAUDE.local.md — 私有笔记（被git忽略）

10.2.2 子Agent并行

错误用法：

  • 把Claude Code当成单线程打工人
  • 做一个任务 → 等 → 再给下一个任务

正确用法：

  • 同时跑5个Agent：安全审计、重构认证、写测试、更新文档、修bug
  • 全部并行执行，共享缓存

10.2.3 权限配置

错误用法：

  • 每次弹出”是否允许这个操作？”都点”是”

正确用法：

  • 在~/.claude/settings.json里配置
  • 用glob模式定义哪些操作默认允许
  • 配置一次，就别再点了

10.2.4 会话管理

错误用法：

  • 每次打开Claude Code，都新开一个session

正确用法：

  • 用--continue
  • 让上下文不断积累
  • 让session memory持续沉淀经验
  • 把/compact当成游戏里的”手动存档点”

10.3 注意事项

  1. 什么不应该保存到记忆：
    • ❌ 代码模式、架构、文件路径 → 可以通过读代码获得
    • ❌ Git历史、最近变更 → git log/git blame是权威来源
    • ❌ 调试方案或修复配方 → 修复在代码里，commit message有上下文
    • ❌ CLAUDE.md中已记录的内容 → 不要重复
    • ❌ 临时任务细节 → 用Task系统，不要写进长期记忆
  2. Memory新鲜度机制：
    • 记忆是时间点快照，不是实时状态
    • 用记忆做起点，但行动前要验证
    • 带行号引用的过期信息反而让错误看起来更权威
  3. 压缩后恢复：
    • 自动注入最近5个文件的最新内容（每文件≤5K token，总预算50K）
    • 已激活的Skills（每个≤5K token，总预算25K）
    • MCP Instructions Delta

11. 总结与启示

11.1 三大支柱

  1. 缓存架构 — 让无限上下文对话在商业上可行
    • 两层缓存（静态/动态分离）
    • 四层递进Compact架构
    • 数百万用户共享同一份静态prompt缓存
  2. 自我进化记忆 — 从”记录对话”到”学习知识”
    • 三层记忆体系（Auto/Team/Agent Memory）
    • Auto Dream（闭环自我进化）
    • 每24小时+累积5个会话后，自动反思
  3. 多模式编排 — 从”被动响应”到”主动执行”
    • Proactive模式（焦点感知+定时唤醒+Skill Discovery+Token Budget）
    • 多Agent协作（Fork/Subagent/Swarm）
    • 用token量驱动的自主工作

11.2 设计哲学

  1. Cache感知：
    • Fork直接复制父级渲染后的system prompt字节
    • Agent根据缓存过期时间（5分钟TTL）决定sleep时长
    • 连Agent的创建方式都要考虑缓存命中率
  2. 基础设施感知：
    • Agent感知缓存TTL、token预算、终端焦点状态
    • 据此调整行为节奏，避免无效唤醒
    • 让Agent理解基础设施的约束，而不是盲目执行

11.3 对AI Agent开发的启示

  1. 上下文管理是核心：
    • 五种压缩策略的存在，本身就说明上下文是瓶颈
    • 能轻量解决的不用重量级方案
    • 压缩后需要自动恢复关键上下文
  2. 并行几乎是免费的：
    • 子Agent共享prompt缓存
    • 5个Agent成本≈1个
    • 这个架构本来就是为并行设计的
  3. 记忆需要分层：
    • 从短期到长期，覆盖不同时间尺度
    • 不仅能”记住”，还能”学习”和”进化”
    • 自动提取、整合、修正知识
  4. 安全是多层次的：
    • 20+道检查流水线
    • 沙箱执行+细粒度权限+网络隔离
    • 每个检查都有唯一ID，触发时上报遥测
  5. 可扩展性是关键：
    • 25+生命周期Hook事件
    • 四种Hook类型
    • Hook系统是真正的扩展API

11.4 结语

Claude Code的核心竞争力不是单个功能，而是这套缓存架构+自我进化记忆+多模式编排的完整基础设施。

特别是cache_edits和分层缓存，是让无限上下文对话在商业上可行的关键。没有这套机制，长对话的成本会让产品无法规模化。

这才是工业级AI Agent的设计哲学。
